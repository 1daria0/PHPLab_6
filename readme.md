# Лабораторная работа №6. Обработка и валидация форм.

## Цель работы.

Освоить основные принципы работы с HTML-формами в PHP, включая отправку данных на сервер и их обработку, включая валидацию данных.

## Тема проекта.

Мини-система `«Тайные признания»`, позволяющая пользователям отправлять анонимные или именные признания, выбирать категорию, указывать дату события, а также просматривать все сохранённые записи в виде таблицы с возможностью сортировки.

## Шаг 1. Определение модели данных.

Модель данных включает следующие поля:

|Поле |Тип |Описание |
|-----|----|---------|
|`id`|`integer` |Уникальный идентификатор|
|`title`|`string` |Заголовок признания (3–100 символов)|
|`content`| `text` |Текст признания (минимум 10 символов)|
|`author`|`string` |Имя автора (макс. 50 символов)|
|`category`	|`enum` |Категория: `love`, `friendship`, `work`, `other`|
|`created_at`|`date` |Дата события (формат YYYY-MM-DD)|
|`is_anonymous`	|`boolean`| Флаг анонимности|
|`created_timestamp`|`integer` |Метка времени создания записи (для сортировки)|

## Шаг 2. Создание HTML-формы.

Форма реализована в файле `form_oop.php`. Она использует метод POST и отправляет данные на `process_oop.php`. Добавлена базовая клиентская валидация с помощью атрибутов:

* `required` – обязательные поля
* `minlength="3"` для заголовка
* `minlength="10"` для текста признания
* `maxlength` для ограничения длины
* `type="date"` для поля даты

## Шаг 3. Обработка данных на сервере.

Обработка реализована в объектно-ориентированном стиле с использованием классов.

Основные классы:

* `Confession` – модель данных (свойства, конструктор, метод `toArray()`).

* `ValidatorInterface` – интерфейс, задающий метод `validate(array $data): array`.

* `ConfessionValidator` – реализация валидации согласно требованиям (проверка длины, формата даты, допустимых категорий).

* `ConfessionStorage` – класс для работы с файлом `data.txt`. Методы:

    * `loadAll()` – загружает все записи из файла, возвращает массив объектов `Confession`.

    * `save(Confession $confession)` – сохраняет новую запись, генерирует ID и `created_timestamp`.

    * `FormHandler` – класс-обработчик: принимает `$_POST`, вызывает валидатор, при успехе сохраняет через хранилище, возвращает результат.

Файл `process_oop.php` – точка входа: создаёт экземпляры, вызывает обработчик, при успехе редирект на `display_oop.php`, при ошибках – сохраняет сообщения в сессию и возвращает на форму.

Все данные сохраняются в файл `data.txt` в формате `JSON` (каждая запись – отдельная строка). Это обеспечивает читаемость и удобство последующего использования.

## Шаг 4. Вывод данных.

Файл `display_oop.php` читает все записи через `ConfessionStorage::loadAll()` и выводит их в HTML-таблицу. Для улучшения пользовательского опыта реализована сортировка:

* Параметры `sort` и `order` передаются через GET-запрос.

* Сортировка выполняется с помощью `usort()` по выбранному полю (доступны: `id`, `title`, `author`, `category`, `created_at`, `created_timestamp`).

* Заголовки таблицы – ссылки, при клике на которые меняются параметры сортировки.

* Вывод данных экранируется через `htmlspecialchars()` для защиты от `XSS`.

* Для поля `category` используются человекочитаемые названия на русском языке.

Также отображается сообщение об успешном добавлении признания (параметр `success`  в URL).

## Шаг 5. Дополнительные функции.

 1. Добавление интерфейса для валидаторов.

Создан интерфейс `ValidatorInterface` с единственным методом `validate(array $data): array`. Класс `ConfessionValidator` реализует этот интерфейс. Это позволяет унифицировать работу валидаторов и легко подключать новые (например, для других сущностей) без изменения основного кода обработки формы.

2. ООП-реализация

Вся логика приложения разделена на отдельные классы по принципу единственной ответственности:

* Модель `(Confession)` – хранит данные и обеспечивает преобразование в массив.

* Валидация `(ConfessionValidator)` – инкапсулирует правила проверки.

* Хранилище `(ConfessionStorage)` – отвечает за чтение и запись в файл.

* Обработчик `(FormHandler)` – координирует процесс: получает данные, валидирует, сохраняет.

* Интерфейс – задаёт контракт для валидаторов.

Такая архитектура повышает читаемость, облегчает тестирование и сопровождение.

## Контрольные вопросы

1. Какие существуют методы отправки данных из формы на сервер? Какие методы поддерживает HTML-форма?

HTML-формы поддерживают два основных метода: `GET` и `POST`.

* `GET` – данные передаются в `URL (query string)`, видны в адресной строке, имеют ограничение по длине (обычно 2048 символов). Используется для запросов, не изменяющих состояние сервера.

* `POST` – данные передаются в теле HTTP-запроса, не видны в URL, не имеют ограничения по длине. Используется для отправки данных, изменяющих состояние (добавление, обновление).

Также современные формы могут использовать методы `PUT`, `DELETE` и другие, но только с помощью `JavaScript` или через атрибут `method` со значением `POST` и скрытым полем` _method`.

2.  Какие глобальные переменные используются для доступа к данным формы в PHP?

Для доступа к данным формы используются суперглобальные массивы:

`$_GET` – данные, переданные методом GET (через URL).

`$_POST` – данные, переданные методом POST (в теле запроса).

`$_REQUEST` – объединяет `$_GET`, `$_POST` и `$_COOKIE` (не рекомендуется из соображений безопасности, так как может содержать неожиданные данные).

Для загружаемых файлов используется массив `$_FILES`.

3. Как обеспечить безопасность при обработке данных из формы (например, защититься от XSS)?

Основные меры безопасности:

* Защита от `XSS` (межсайтового скриптинга): при выводе данных, полученных от пользователя, всегда использовать `htmlspecialchars()` для экранирования HTML-тегов.

* Защита от `SQL-инъекций`: если используется база данных, применять подготовленные выражения `(PDO, MySQLi)`.

* Валидация и санитизация: проверять данные на сервере на соответствие ожидаемым типам, диапазонам, форматам. Использовать фильтры `(filter_input)`.

* Защита от `CSRF` (межсайтовой подделки запроса): использовать токены в формах.

* Ограничение размера загружаемых данных: настройки `post_max_size` и `upload_max_filesize в php.ini`.

* Использование `HTTPS`: для шифрования передаваемых данных.
